Never add new UI/system messages on resume

Do not add any message like “CHAT RESTORED”, “Session resumed”, banners, or any admin/system notice to the chat UI.

The chat UI after reload must show exactly the same messages that existed before reload, in the same order, with the same roles (user/assistant/system). No extras.

Assessment reload invariants (must be explicit in plan)

A1 (reload before answer): If reload occurs before the user answers the current MCQ, resume by rendering the full prior chat and show the same current question as the last message, waiting for the user’s answer. Do not regenerate or rephrase the question. Do not generate a different question at the same level.

A2 (reload right after answer while next MCQ is being generated): On resume, render the full prior chat including the user’s last answer, and the next MCQ that was already being generated must appear as the next assistant message. Do not produce a new or duplicate question.

Intake behavior (one-time only, with precise persistence rules)

Intake prompts/messages appear only once in a user’s life cycle (first-ever session after signup) and are saved to chat_history only during that first session.

If a reload happens during that first intake session, the intake messages are restored and the user can finish intake normally.

After intake is completed and saved in profile/dashboard, intake must never appear again for this user in any new session. New sessions must start directly with the welcome message and then assessment, with chat persistence from that point onward.

Teaching session autosave (no empty sessions)

From the moment teaching starts, every message (assistant or user) must be saved immediately to chat_history and to the persistent teaching session record. Do not rely on “End Lesson & Save” for persistence; it only ends the lesson, not the saving.

Reload during teaching must restore the entire conversation and continue seamlessly. No empty teaching sessions must ever be recorded.

active_sessions table: id (uuid pk), user_id, session_id (unique), stage, chat_history (jsonb, default []), assessment_state (jsonb), teaching_state (jsonb), updated_at (timestamptz, default now()).

POST /api/intake/next: Append intake messages to chat_history only during the first-ever intake session. Respect the one-time rule.

POST /api/assess/next and POST /api/assess/answer:

Append messages to chat_history.

Maintain assessment_state flags exactly (awaiting_user_answer and is_generating_next).

Never start generating a new question if awaiting_user_answer=true or is_generating_next=true.

Move stage to 'assessment' when appropriate.

POST /api/report: Append the report to chat_history and set stage='report'.

POST /api/teach/start, POST /api/teach/message, POST /api/teach/save:

Set stage='teaching' at start.

POST /api/session/clear: Clear active session or mark stage='closed'. New sessions will show welcome only after this endpoint is called.
