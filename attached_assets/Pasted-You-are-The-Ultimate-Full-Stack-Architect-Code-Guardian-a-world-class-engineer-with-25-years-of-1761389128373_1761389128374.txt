Fix the chat reloading (rehydration) and session management system across all phases — Assessment, Report, and Teaching — so that the user’s chat state, messages, and progress are fully preserved through reloads or page restarts, without introducing any new errors or affecting other parts of the system.

Current Problems
Assessment Phase (Mid-session reload):
When the user reloads the chat page during an ongoing assessment, the backend correctly remembers progress (e.g., which question the user is on),
but the frontend clears the chat window, adds a “Welcome” message, and then re-fetches the last question.
This causes the UI to look as if the session restarted, even though the data is still correct internally.

Report & Teaching Phase (Reloads):

Reloading during or after report generation wipes the chat and starts a new assessment, instead of restoring the report view or ongoing teaching conversation.
During teaching, if a reload happens, empty “Teaching Session” entries are created in the dashboard.

UI & Session Issues:

Some system messages like  “Welcome!” appear after reload — these are unwanted.
The button “END LESSON & SAVE” must be completely removed.
Teaching sessions must be saved only when the user explicitly presses “NEW ASSESSMENT”, never before.

New Assessment Session:

Starting a new assessment should create a completely new chat context (new attempt, new teaching thread).
The new teaching session must not carry over context or memory from the previous one.
✅ Expected Behavior / Requirements
1. Mid-Assessment Reload

On page reload during an active assessment:
The system must restore the exact same chat transcript (both user and assistant messages).
If a question was on screen but not yet answered → it remains exactly as it was.
If the user answered but the next question wasn’t yet generated → show the last answer, then continue naturally.
Do not show any system messages such as “Welcome!” or “Chat restored.”

2. Post-Report Reload

When the report is shown and the user reloads the page:
The report reappears exactly as before.
No new assessment starts automatically.
The user can press “Start Explanation” to begin teaching as usual.

3. Mid-Teaching Reload

If the user reloads while waiting for the model’s reply → the chat should resume and automatically show the pending message once it’s available.
If the user reloads while waiting to type a reply → the last tutor message and input box should appear exactly as before.
No empty “Teaching Session” entries should appear in the dashboard in any of these cases.

4. Teaching Session Storage & UI

Remove the title “Teaching Session” entirely from the chat UI.
Completely remove the button “END LESSON & SAVE.”
Teaching explanations must be saved only when the user clicks “NEW ASSESSMENT.”

5. Starting a New Assessment

“NEW ASSESSMENT” should:
Fully clear the chat window and client state.
Close any open attempt or teaching thread.
Start a completely new session (new attempt ID, new teaching thread).
Ensure no previous teaching context leaks into the new one.

6. Stability & Reliability

The entire system should handle reloads and restores safely —
No duplicated questions.
No repeated OpenAI API calls on reload.
No partial rehydration (must include both user and assistant messages).
The user experience should feel identical to never having reloaded.

 Strict Prohibitions

❌ No “Welcome”, “Chat restored”, or any other system/debug messages after reload.
❌ No partial transcript restores (must include both sides).
❌ No auto-creation of Teaching Session rows in the database on reload or start.
❌ No saving of teaching data unless “NEW ASSESSMENT” is pressed.
❌ No reuse of old threads or assessment IDs across new sessions.

 Implementation Notes (Recommended Technical Direction)

Create a unified API endpoint /api/chat/state (or reuse existing logic) to return:
Current phase (intake, assessment, report, teaching)
Active IDs (attemptId, threadId)
Complete chat transcript (all messages)
Pending message states (waiting for model/user)
Run status for teaching threads (in_progress, completed)

On page load (chat.js):

Call /api/chat/state instead of resetting chat.
Re-render full chat transcript exactly as before.
Do not insert any new messages or greetings.
If a run is still in progress → resume polling until response arrives.
Prevent duplicate teaching records:
Create teaching_notes only after pressing NEW ASSESSMENT.
Optionally store in-progress teaching data as draft in memory, not DB.
Add Idempotency Keys for /api/assess/answer, /api/teach/start, /api/teach/message
to prevent duplicate API calls if the user reloads or the network retries.

Ensure NEW ASSESSMENT endpoint:
Closes active attempt (if any).
Deletes local chat state.
Starts clean new session with new IDs.
