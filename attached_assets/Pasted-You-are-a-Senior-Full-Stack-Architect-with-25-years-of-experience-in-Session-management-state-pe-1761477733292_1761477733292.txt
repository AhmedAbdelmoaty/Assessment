THE PROBLEM (Current Behavior)
Critical Issue

When a user reloads the page or navigates away and returns, the chat interface completely resets and loses all previous messages, even though the backend session is partially preserved.
Scenario 1: Reload During Assessment (MCQ Questions)
Current (Broken) Behavior:
User is on Question 2 of Level 2 (L2).
User reloads the page.

Chat UI clears.
System shows: “مرحبًا! هنبدأ أسئلة التقييم مباشرة” (Welcome message).
Then displays Question 2 of L2 as the first message in an empty chat.
All previous questions and answers are gone.

Expected Behavior:
User reloads while on Question 2 of L2.
Chat UI shows all previous questions and answers.
Last message is Question 2 of L2, waiting for an answer.
No welcome message.
Flow continues seamlessly as if reload never happened.

Detailed Continuity Requirements:

A1) Reload before answering the current question:
On resume, show the entire previous conversation.
The current question must appear exactly as it was and wait for the user’s answer.
Do not regenerate a new question or rephrase the existing one.
The state remains “awaiting user answer.”

A2) Reload after the user has answered and the next question is being generated:
On resume, show the full chat history including the last user answer.
The next question that was being generated must appear as the next assistant message.
Do not generate a new or duplicate question.
Continue the generation result that was already in progress.

Scenario 2: Reload After Report Generation
Current (Broken) Behavior:
User finishes the assessment and sees the report.
User reloads → Chat clears → New assessment starts.

Expected Behavior:
On reload, chat shows the full conversation including the report.
Last message is the report itself.
User can click “Start Explanation” to continue.
No new assessment or reset.

Scenario 3: Reload During Teaching Session
Current (Broken) Behavior:
User is in teaching mode chatting with the AI tutor.
User reloads → Chat clears → “Welcome” appears → new empty teaching session created.

Expected Behavior:

Reload → Chat shows the full conversation (report + teaching messages).
Last message is the tutor’s response or the user’s last question.
The tutor maintains full context from the OpenAI thread.
No empty teaching sessions saved in the dashboard.

REQUIRED OUTCOME

The chat must behave exactly like Claude or ChatGPT:
Reloading shows all prior messages and context.
The conversation continues seamlessly from the exact point it was left.
No context or messages are lost.

Critical Rules

Chat messages must persist across reloads.
Session state (assessment progress, teaching context, report) must persist.
“Welcome” message only appears for brand-new sessions (via “New Assessment” button).
Chat resumes exactly as before — same state, same waiting/question logic.
No empty teaching sessions saved to the database.
In assessments, follow A1/A2 invariants exactly.

Phase 2: Backend APIs
Modify routes to fully persist and restore session state:
/api/session/state
If logged in → fetch user’s active session from DB or create new.
Return { sessionId, stage, chatHistory, assessmentState, teachingState }.
/api/intake/next
When starting → save sessionId + stage='intake'.
Append all intake messages to chat_history.
/api/assess/next and /api/assess/answer
Append messages and answers to chat_history.
Persist assessment_state including:
awaiting_user_answer (true/false)
is_generating_next (true/false)
current_question (full question object)
next_question_buffer (optional generated question waiting)
Update these flags on each transition.
Prevent new question generation if awaiting_user_answer=true or is_generating_next=true.
/api/report
Append report text to chat_history.
Set stage = 'report'.
/api/teach/start, /api/teach/message, /api/teach/save
Append all messages to chat_history.
Save teaching_state.
Auto-save after every message.
Set stage = 'teaching'.
/api/session/clear
Optional: clear user’s active session or set stage='closed'.
