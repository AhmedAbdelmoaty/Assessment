Fix remaining issues in chat restoration: double init, auto-start on reload, and teaching context leakage

Hi! Thanks for the previous pass. A lot of the core issues are still present. Here’s what I’m seeing and what needs to be fixed precisely.

What I’m observing now (bugs)

Duplicate rendering: On load and on reload, the same question (and sometimes the whole block) appears twice.

Auto-restart on reload: The chat still clears and restarts with greeting-like messages after a reload, instead of restoring the exact transcript.

Incomplete restoration: The full transcript is not restored 1:1 (both user and assistant messages, same order, no extra messages).

Teaching context leakage: After pressing NEW ASSESSMENT, the new teaching session still continues with context from the previous one.

These indicate:

Multiple boot paths (double initialization) in public/chat.js.

Auto-start logic is still executed on reload (greetings / startAssessment).

/api/chat/state might be doing more than read-only or returning a fallback phase that makes the UI start fresh.

Teaching threadId (or similar) is not being cleared on NEW ASSESSMENT.

What to change (do exactly this)
A) Frontend (public/chat.js)

Single bootstrap only

Add a single DOMContentLoaded handler that does await init() once using a bootstrapped flag.

Remove/disable any other code that calls init() or checkSessionState() on load (window.onload, direct calls, etc.).

bootstrapped example:

let bootstrapped = false;
document.addEventListener("DOMContentLoaded", async () => {
  if (bootstrapped) return;
  bootstrapped = true;
  await init();
});


No auto-start on reload

In checkSessionState():

Fetch /api/chat/state.

Set sessionId, currentStep.

Rebuild the transcript ONLY from the returned transcript.

Do NOT add any greeting/welcome message.

Do NOT call startAssessment() or any “begin flow” logic from checkSessionState().

Rehydration

Implement a rehydrateTranscript(items) that:

Clears the message container.

Renders each item in order (user → user bubble, assistant → assistant bubble).

Uses a guard isRehydrating so that we do not POST these messages back into /api/chat/track while rebuilding.

In addUserMessage / addSystemMessage, only POST to /api/chat/track when !isRehydrating.

Search & remove any leftover greeting or auto-start code

Remove any occurrences of “Welcome”, “مرحبًا”, “Chat restored”, or any call to startAssessment() that runs on page load / reload.

New Assessment isolation (frontend)

The NEW ASSESSMENT button must not reuse any previous state.

It should clear the UI and then call the backend endpoint that starts a fresh session.

Do not carry over local variables that hold old thread/session info.

B) Backend (server/index.js)

/api/chat/state must be read-only

It should not create a new session, not reset transcript, not change currentStep.

It should only return:

{ ok: true, sessionId, phase, transcript: [...], teaching: { threadId, runStatus } }

Make sure phase does not fall back to "intake" when there is an active assessment/report/teaching, and that nothing in this route triggers any new question generation or greeting message.

No auto-generation side effects

Verify that no other endpoints called during load/reload generate a new question automatically unless explicitly requested by the user flow (e.g., after answering).

New Assessment isolation (backend)

On the endpoint that starts a new assessment:

Close/finish the previous attempt if open.

Clear any teaching data for the previous run (threadId, draft state, etc.).

Reset the UI transcript for the new session.

Ensure a new assessment id and a new teaching thread will be used (no reuse).

Teaching thread context

Ensure that after NEW ASSESSMENT, the next teaching session uses a new thread and has no access to the previous thread/messages. No leakage.

Acceptance tests (must all pass)

Mid-assessment reload: Reload while a question is visible and unanswered → the exact same single question and full transcript appear, no greetings, no duplicates.

Answer then reload before next: After sending answer but before next question appears → on reload I see my last answer and then the next question appears naturally; no greetings.

Post-report reload: When report is visible, reload → I see the report again as-is; no new assessment starts.

Mid-teaching reload: While waiting for assistant’s reply, reload → the conversation is fully restored; no empty teaching entries; when the assistant’s reply is ready it appears.

New assessment isolation: After pressing NEW ASSESSMENT, the new session is clean. Later, when I start teaching, the tutor does not continue from previous explanations; it’s a fresh thread.

No double init: checkSessionState() runs exactly once on load; no duplicated rendering.

No unexpected system messages: No “Welcome”, “Chat restored”, or any extra system/debug messages appear on reload.

Please implement these changes and verify each acceptance test. If something in the current code prevents a clean single-boot path, point me to the exact lines and propose the minimal safe change.
